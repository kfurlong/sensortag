    50    /*!
    51     *  ======== Load ========
    52     *  The Load module reports execution times and load information for threads
    53     *  in a system.
    54     *
    55     *  SYS/BIOS manages four distinct levels of execution threads: hardware
    56     *  interrupt service routines, software interrupt routines, tasks, and
    57     *  background idle functions. This module reports execution time and load on a
    58     *  per-task basis, and also provides information globally for hardware
    59     *  interrupt service routines, software interrupt routines and idle functions
    60     *  (in the form of the idle task). It can also report an estimate of the
    61     *  global CPU load.
    62     *
    63     *  Execution time is reported in units of
    64     *  {@link xdc.runtime.Timestamp Timestamp} counts, and load is reported
    65     *  in percentages.
    66     *
    67     *  By default, load data is gathered for Task threads.
    68     *  {@link #hwiEnabled hwiEnabled}, {@link #swiEnabled swiEnabled}
    69     *  and {@link #taskEnabled taskEnabled} can be used to select which
    70     *  type(s) of threads are monitored. These statistics are automatically
    71     *  recorded to the load module's logger instance. Users can also choose
    72     *  to call {@link #getTaskLoad}, {@link #getGlobalSwiLoad},
    73     *  {@link #getGlobalHwiLoad} and {@link #getCPULoad} at any time to obtain the
    74     *  statistics at runtime.
    75     *
    76     *  The module relies on {@link #update} to be called to compute load and
    77     *  execution times from the time when {@link #update} was last called. This
    78     *  is automatically done for every period specified by
    79     *  {@link #windowInMs windowInMs} in an {@link ti.sysbios.knl.Idle Idle}
    80     *  function when {@link #updateInIdle updateInIdle} is
    81     *  set to true. The time between 2 calls to {@link #update} is called the
    82     *  benchmark time window.
    83     *
    84     *  By passing in a function pointer of type
    85     *  Void(*{@link #postUpdate})(Void) through the
    86     *  {@link #postUpdate postUpdate} config parameter, one can specify a
    87     *  {@link #postUpdate postUpdate} function that is
    88     *  automatically called by {@link #update} immediately after the statistics
    89     *  have been computed at the end of a benchmark time window.  Setting this
    90     *  function allows the user to optionally post-process the thread statistics
    91     *  once and only once per benchmark window.
    92     *
    93     *  Advanced users could optionally omit calling {@link #update} in the idle
    94     *  loop by setting {@link #updateInIdle updateInIdle} to false. They can
    95     *  then call {@link #update} somewhere else (e.g. in a periodic higher
    96     *  priority Task) instead to ensure statistics are computed even when the
    97     *  system is never idle.
    98     *
    99     *  @a(CPU Load Calculation Methods)
   100     *
   101     *  The CPU load is computed in three different ways,
   102     *  depending on what threads are monitored, and whether or not Power
   103     *  management is used to idle the CPU when no threads are running.
   104     *
   105     *  @p(html)
   106     *  <B>Task Load Disabled and No Power Management</B>
   107     *  @p
   108     *
   109     *  The first method of calculating CPU load is used when
   110     *  Task load logging is disabled, ie, {@link #taskEnable}
   111     *  is false, and Power management is not used. The CPU load is computed as
   112     *  the percentage of time in the benchmark window which was NOT spent in the
   113     *  idle loop. More specifically, the load is computed as follows:
   114     *
   115     *  global CPU load = 100 * (1 - (min. time for a trip around idle loop *
   116     *      # times in idle loop)/(benchmark time window) )
   117     *
   118     *  Any work done in the idle loop is included in the CPU load - in other
   119     *  words, any time spent in the loop beyond the shortest trip around the idle
   120     *  loop is counted as non-idle time.
   121     *
   122     *  This method works fairly well if the timestamp frequency is
   123     *  sufficiently high (for example, if it's equal to the CPU frequency).
   124     *  For the MSP430, however, with the CPU running at 8MHz, when the 32KHz
   125     *  ACLK is used for the timestamp counter, the CPU load is only a very
   126     *  rough approximation, due to the courseness of the timestamp.  The CPU load
   127     *  accuracy can also be affected by caching and user idle functions.
   128     *
   129     *  @p(html)
   130     *  <B>Task Load Enabled and No Power Management</B>
   131     *  @p
   132     *
   133     *  The second method of calculating CPU load is used when Task load logging
   134     *  is enabled ({@link #taskEnable} = true) and Power management is not used.
   135     *  In this case the CPU load is calculted as
   136     *
   137     *      global CPU load = 100 - (Idle task load)
   138     *
   139     *  This prevents any discrepancy between the calculated CPU load had we
   140     *  used the first method, and 100 - the Idle task load.  If Swi and
   141     *  Hwi load logging are not enabled, however, time spent in a Swi
   142     *  or Hwi will be charged to the task in which it ran.  This will
   143     *  affect the accuracy of the CPU (and Task) load, but the trade off
   144     *  is more overhead to do the Hwi and Swi load logging.
   145     *
   146     *  @p(html)
   147     *  <B>Power Management Enabled</B>
   148     *  @p
   149     *
   150     *  The third method of calculating CPU load is used when Power
   151     *  management is enabled.  In this case, the idle loop has a
   152     *  function brought in by the Power module that idles the CPU.
   153     *  The timestamp timer must continue to run during idle, in order
   154     *  to measure idle and non-idle time, however.  This method of
   155     *  calculating CPU load plugs a Hwi hook function which is run
   156     *  at the beginning of every Hwi. An idle function will call
   157     *  Timestamp_get32() to mark the beginning of idle time, and
   158     *  when the first Hwi causes the CPU to come out of idle, the
   159     *  Hwi hook function will call Timestamp_get32() to mark the end
   160     *  of idle time.  The idle function accumulates the idle time
   161     *  and total time elapsed, and the CPU load is calculated as:
   162     *
   163     *    global CPU load = 100 * (1 - idle time / time elapsed)
   164     *
   165     *  With this method of CPU load calculation, all idle functions are
   166     *  included in the CPU load.  When Power is enabled, Task, Swi, and
   167     *  Hwi load logging can be enabled, but you may find that the
   168     *  CPU load does not equal (100 - idle task load), since the CPU load
   169     *  is calculated as the percentage of time that the processor is not
   170     *  powered down, while the idle task load includes powered down time
   171     *  plus time executing idle functions.
   172     *
   173     *  @a(Examples)
   174     *  Configuration example: The following statements configure the Load module
   175     *  to obtain output in the CCS output window. This is useful for early
   176     *  development without RTA tools support in SYS/BIOS:
   177     *
   178     *  @p(code)
   179     *  var System = xdc.useModule('xdc.runtime.System');
   180     *  var SysStd = xdc.useModule('xdc.runtime.SysStd');
   181     *  System.SupportProxy = SysStd;
   182     *
   183     *  var Load = xdc.useModule('ti.sysbios.utils.Load');
   184     *  var LoggerSys = xdc.useModule('xdc.runtime.LoggerSys');
   185     *  var Defaults = xdc.useModule('xdc.runtime.Defaults');
   186     *  var Diags = xdc.useModule('xdc.runtime.Diags');
   187     *
   188     *  // Turn on statistics log messages for Load module
   189     *  Load.common$.diags_USER4 = Diags.ALWAYS_ON;
   190     *  // Define a program-wide logger to be the System Logger
   191     *  Defaults.common$.logger = LoggerSys.create();
   192     *  @p
   193     *
   194     *  Notes on above example:
   195     *  @p(blist)
   196     *  - When using the {@link xdc.runtime.LoggerSys System Logger},
   197     *  {@link xdc.runtime.System#printf System_printf} calls are made to produce
   198     *  the output in the output window. These calls are highly intrusive
   199     *  and are only suited for early development.
   200     *
   201     *  - Also make sure your task stack sizes are large enough when using the
   202     *  {@link xdc.runtime.LoggerSys System Logger} - particularly the idle task
   203     *  stack size since logging is done in the idle task by default.
   204     *  @p
   205     *
   206     *  @a(Caveats)
   207     *  @p(nlist)
   208     *  - For the module to return accurate load values, the {@link #update}
   209     *  function must be run at least once before the Timestamp count gets to
   210     *  wrap around. This means on a platform with a 32-bit
   211     *  {@link xdc.runtime.Timestamp Timestamp} frequency that runs at 200 MHz,
   212     *  the function must be called at least once every 21 sec. With faster
   213     *  {@link xdc.runtime.Timestamp Timestamp} frequencies it would have to be
   214     *  called even more frequently.
   215     *
   216     *  - If the {@link #updateInIdle updateInIdle} option is enabled, the
   217     *  module will call {@link #update} periodically according to
   218     *  {@link #windowInMs windowInMs}. The latter must be set to a time
   219     *  interval that is below
   220     *  (2^32) * 1000 / ({@link xdc.runtime.Timestamp Timestamp} Frequency)
   221     *  milliseconds
   222     *
   223     *  - If the {@link #updateInIdle updateInIdle} option is disabled, the user is
   224     *  then responsible for calling {@link #update} more often than the above
   225     *  time interval.
   226     *
   227     *  - Load values might not add up to 100%. Because the module minimizes
   228     *  its interrupt latency and avoids locking up the system to compute all
   229     *  thread loads at once, each thread's execution time is evaluated separately
   230     *  over its own time window, which could be slightly different to that of
   231     *  another thread.
   232     *
   233     *  - When disabling monitoring of a thread type of higher priority, the
   234     *  time spent in that thread type will be counted towards time spent in
   235     *  threads of lower priority. E.g. if {@link #hwiEnabled hwiEnabled} and
   236     *  {@link #swiEnabled swiEnabled} are both false, then time spent in Hwi's and
   237     *  Swi's will be counted towards the Tasks in which the interrupts happened.
   238     *  Thus, for better accuracy, it is best to leave monitoring on for threads of
   239     *  a higher priority relative to the thread type of interest.
   240     *
   241     *  - When Task load logging is not enabled and Power management is not used,
   242     *  the implementation of {@link #getCPULoad()} self-calibrates the shortest
   243     *  path through the idle loop. It does this by keeping track of the shortest
   244     *  time between invocations of an idle function automatically inserted by the
   245     *  Load module, and assumes that to be the time it takes for one iteration
   246     *  through the idle loop. Because of this, the CPU load value is only an
   247     *  estimate since the idle loop might occasionally take longer to run
   248     *  (e.g. due to caching effects, stalls). The reported CPU load tends to be
   249     *  slightly higher than reality, especially when the load is low.
   250     *
   251     *  - If {@link #taskEnabled taskEnabled} is set to true,
   252     *  task name support will be automatically turned on so that the Load module
   253     *  can print out the task names in its log output.
   254     *
   255     *  - Currently does not support {@link xdc.runtime.Timestamp Timestamp}
   256     *  frequencies over 4 GHz.
   257     *  @p
   258     *
   259     *  @p(html)
   260     *  <h3> Calling Context </h3>
   261     *  <table border="1" cellpadding="3">
   262     *    <colgroup span="1"></colgroup> <colgroup span="5" align="center">
   263     *    </colgroup>
   264     *
   265     *    <tr><th> Function                 </th><th>  Hwi   </th><th>  Swi   </th>
   266     *    <th>  Task  </th><th>  Main  </th><th>  Startup  </th></tr>
   267     *    <!--                                                          -->
   268     *    <tr><td> {@link #getCPULoad}      </td><td>   Y    </td><td>   Y    </td>
   269     *    <td>   Y    </td><td>   N    </td><td>   N    </td></tr>
   270     *    <tr><td> {@link #getGlobalHwiLoad}</td><td>   Y    </td><td>   Y    </td>
   271     *    <td>   Y    </td><td>   N    </td><td>   N    </td></tr>
   272     *    <tr><td> {@link #getGlobalSwiLoad}</td><td>   Y    </td><td>   Y    </td>
   273     *    <td>   Y    </td><td>   N    </td><td>   N    </td></tr>
   274     *    <tr><td> {@link #getTaskLoad}     </td><td>   Y    </td><td>   Y    </td>
   275     *    <td>   Y    </td><td>   N    </td><td>   N    </td></tr>
   276     *    <tr><td> {@link #reset}           </td><td>   Y*   </td><td>   Y*   </td>
   277     *    <td>   Y    </td><td>   Y    </td><td>   N    </td></tr>
   278     *    <tr><td> {@link #update}          </td><td>   Y*   </td><td>   Y*   </td>
   279     *    <td>   Y    </td><td>   N    </td><td>   N    </td></tr>
   280     *    <tr><td colspan="6"> Definitions: <br />
   281     *       <ul>
   282     *         <li> <b>Hwi</b>: API is callable from a Hwi thread. </li>
   283     *         <li> <b>Swi</b>: API is callable from a Swi thread. </li>
   284     *         <li> <b>Task</b>: API is callable from a Task thread. </li>
   285     *         <li> <b>Main</b>: API is callable during any of these phases: </li>
   286     *           <ul>
   287     *             <li> In your module startup after this module is started
   288     *    (e.g. Load_Module_startupDone() returns TRUE). </li>
   289     *             <li> During xdc.runtime.Startup.lastFxns. </li>
   290     *             <li> During main().</li>
   291     *             <li> During BIOS.startupFxns.</li>
   292     *           </ul>
   293     *         <li> <b>Startup</b>: API is callable during any of these phases:</li>
   294     *           <ul>
   295     *             <li> During xdc.runtime.Startup.firstFxns.</li>
   296     *             <li> In your module startup before this module is started
   297     *    (e.g. Load_Module_startupDone() returns FALSE).</li>
   298     *           </ul>
   299     *       <li> <b>*</b>: Indicates only when {@link #taskEnabled taskEnabled}
   300     *    is set to false. </li>
   301     *       </ul>
   302     *    </td></tr>
   303     *  </table>
   304     *  @p
   305     */

   513        /* -------- Module Functions -------- */

   515        /*!
   516         *  ======== getTaskLoad ========
   517         *  Return the load and time spent in a specific task
   518         *
   519         *  This function returns the load and time spent in a specific task
   520         *  along with the duration over which the measurement was done. Numbers
   521         *  are reported in  {@link xdc.runtime.Timestamp Timestamp} counts.
   522         *
   523         *  Task handle must be valid and have been registered with Load.
   524         *
   525         *  @param(task) Handle of the Task which time we are interested in.
   526         *  @param(stat) Load and time statistics info
   527         *  @b(returns) TRUE if success, FALSE if failure
   528         *
